# ТЗ по полным связям между ролями и полями проекта

## 1. Цель

Сделать проект единым контуром данных, где все роли работают не с разрозненными локальными списками, а с общей базой сущностей.

Главный принцип:

- если в форме есть поле `Гражданин`, `Человек`, `ФИО`, `Отец`, `Мать`, `Ребенок`, `Семья`, `Учреждение`, `Паспорт`, `Адрес`, `Военная запись`, `Медзапись`, то это поле должно быть связано с общей базой
- выбор должен идти по `ID`, а отображение по `ФИО/названию`
- вручную вводить текст вместо связи можно только как временное исключение, когда объект еще не создан или не найден

## 3. Главная проблема проекта сейчас

После изучения проекта видно несколько системных проблем:

### 3.1 Смешаны разные идентификаторы

Сейчас используются оба названия:
- `citizenId`
- `peopleId`

По факту это почти всегда один и тот же человек.

Требование:
- на backend оставить единый смысл: это ссылка на `Citizen.id`
- на frontend в ТЗ принять правило:
  - `peopleId` и `citizenId` должны означать одно и то же
  - в новых формах желательно привести к одному имени, лучше `citizenId`
  - если backend пока не меняется, то на уровне UI и mapping явно документировать, что `peopleId = Citizen.id`

### 3.2 Часть модулей уже работает через backend, часть еще живет в localStorage

Критичная проблема найдена в `Погранслужбе`:
- `border-crossing-create-edit` берет людей не из backend `citizens`
- а из `localStorage`, `school`, `maternity`, `workflow`

Из-за этого поле `Гражданин` там неполное или пустое.

Требование:
- `Погранслужба` должна брать граждан только из общего реестра `citizens`
- локальные источники должны быть убраны из выбора граждан

### 3.3 Есть текстовые поля, которые должны быть ID-связями

Сейчас во многих местах есть одновременно:
- `fatherFullName`
- `motherFullName`
- `childFullName`

Это допустимо только как отображение или резервная копия текста.

Требование:
- главная связь всегда по `fatherCitizenId`, `motherCitizenId`, `childCitizenId`
- поля `*FullName` должны быть производными и автозаполняемыми

## 4. Базовые правила связности по всему проекту

### 4.1 Поле `Гражданин`

Если в форме есть поле:
- `Гражданин`
- `Человек`
- `ФИО`
- `peopleId`
- `citizenId`

То оно должно:
- загружаться из `GET /api/citizens`
- сохранять в запись именно `Citizen.id`
- отображать `fullName`
- при выборе гражданина подтягивать связанные данные

Что подтягивать автоматически:
- дата рождения
- пол
- гражданство
- отец
- мать
- семья
- активный адрес
- паспорт, если есть
- военная запись, если есть

### 4.2 Поля `Отец` и `Мать`

Если в форме есть:
- `Отец`
- `Мать`
- `fatherCitizenId`
- `motherCitizenId`

То они должны:
- выбираться из общего списка граждан
- сохранять ID родителя
- текст `fatherFullName`, `motherFullName` должен ставиться автоматически

Допустимый ручной ввод:
- только если родитель еще не заведен в системе
- тогда запись должна быть временно текстовой и помечаться как `требует связывания`

### 4.3 Поле `Семья`

Если в форме есть:
- `familyId`
- `Семья`

То поле должно:
- выбираться из `GET /api/families`
- при выборе семьи подтягивать:
  - отца
  - мать
  - детей
  - адрес семьи
  - семейную льготу

### 4.4 Поле `Учреждение`

Если есть:
- `institutionId`

То источник:
- `GET /api/education-institutions`

Нельзя:
- вводить название учреждения вручную, если оно уже есть в справочнике

### 4.5 Поле `Пользователь`

Если запись создается сотрудником роли:
- не надо вручную выбирать пользователя в форме
- должен сохраняться `currentUser.id`
- на UI можно показывать только кто создал запись

### 4.6 Поле `Организация`

Если есть:
- `organizationId`

То источник:
- общий справочник `organizations`

Если организация определяется по роли:
- должна подставляться автоматически

## 5.1 Роддом

Сущность:
- `maternity_records`

Главные поля:
- `childCitizenId`
- `motherCitizenId`
- `fatherCitizenId`
- `familyId`
- `birthCaseType`
- `birthDateTime`
- `placeOfBirth`
- `gender`
- `birthWeight`

### Что должно быть связано

#### При создании

Поля:
- `Отец` -> из `citizens`
- `Мать` -> из `citizens`
- `Семья` -> автопоиск по отцу/матери

Автодействия:
- если ребенок новый, создается `Citizen`
- в `Citizen` проставляются:
  - `birthDate`
  - `gender`
  - `fatherCitizenId`
  - `motherCitizenId`
  - `familyId`
- если ребенок мальчик:
  - создается первичная запись в `military_records`
  - `militaryRegisteredAtBirth = true`
- если ребенок девочка:
  - в военкомат не отправлять
- запись автоматически должна быть доступна в:
  - ЗАГС
  - ЖЭК/семьи
  - Военкомат

#### При редактировании

Если меняется:
- отец
- мать
- пол
- дата рождения

То должны пересчитаться:
- семья
- связь ребенка с отцом и матерью
- число детей у отца
- семейная льгота отца
- первичный воинский учет ребенка, если это мальчик

#### При показе

Должно отображаться:
- ссылка на `Citizen`
- ссылка на `Family`
- статус передачи в ЗАГС
- статус постановки в военкомат

Типы:

### Что должно быть связано

#### Рождение

Поля:
- `maternityRecordId` -> из `maternity_records`
- `childCitizenId` -> из `citizens`
- `fatherCitizenId` -> из `citizens`
- `motherCitizenId` -> из `citizens`

Автодействия:

Поля:
- `spouseOneCitizenId`
- `spouseTwoCitizenId`

Автодействия:
- создать/обновить `family`
- обоим супругам поставить одинаковый `familyId`
- будущие дети должны попадать в эту семью

#### Смерть

Поля:
- `deceasedCitizenId`

Автодействия:
- обновить `Citizen.lifeStatus`
- если умер отец/мать:
  - обновить семейную модель
- если умер гражданин на воинском учете:
  - военная карточка должна получить финальный статус

## 6. Что должно происходить при создании, редактировании и показе

### 6.1 При создании записи

Общее правило:
- пользователь выбирает связанный объект из справочника
- сохраняется `ID`
- связанные текстовые поля подставляются автоматически
- если создание влияет на другой модуль, обновление идет автоматически

Примеры:
- рождение -> гражданин + семья + военкомат
- паспорт -> карточка гражданина + погран + военкомат
- учеба -> отсрочка в военкомате
- ВВК -> категория годности в военкомате
- граница -> статус поездки в карточке гражданина

### 6.2 При редактировании записи

Если меняется ключевое связанное поле:
- `citizenId/peopleId`
- `fatherCitizenId`
- `motherCitizenId`
- `familyId`
- `institutionId`
- `medicalRecordId`

То должны обновляться и зависимые связи.

Нельзя:
- оставлять старые текстовые значения без пересчета

### 6.3 При показе записи

Каждый реестр должен показывать не просто свою строку, а минимум:
- связанного гражданина
- семью
- адрес
- источник создания
- связанные документы/статусы

## 8. Технический долг, найденный при изучении проекта

### Критичный

1. `Погранслужба` использует `localStorage`, а не backend как основной источник людей.
2. В проекте смешаны `peopleId` и `citizenId`.
3. Часть связей дублируется текстом через `FullName`, а не через `ID`.
4. Есть локальный `LocalPersonWorkflowService`, который временно склеивает статусы вне основной базы.

### Средний

1. Не все формы используют единый справочник граждан.
2. Не везде автоматом подтягиваются родители, адрес, паспорт, семья.
3. Не все экраны показывают связанные записи гражданина.

## 9. Что надо реализовать следующим этапом

### Этап 1. Нормализация связей

- везде принять `Citizen` как главный справочник
- везде подключить поле `Гражданин` к общему API
- убрать локальные списки людей

### Этап 2. Нормализация идентификаторов

- унифицировать `peopleId/citizenId`
- на frontend и backend ввести единое правило именования

### Этап 3. Автоподтягивание зависимых полей

- отец
- мать
- семья
- адрес
- паспорт
- учеба
- медстатус

### Этап 4. Автосинхронизация бизнес-логики

- рождение -> семья/военкомат
- учеба -> отсрочка
- отчисление -> возврат в контроль военкомата
- ВВК -> годность
- граница -> статус выезда

## 10. Короткий итог простым языком

Проект должен работать так:

- один раз создается `Гражданин`
- дальше все роли работают с этим же гражданином
- каждая новая запись не дублирует человека текстом, а ссылается на него по ID
- если где-то есть поле `Гражданин`, там должен быть весь список граждан из базы
- если где-то есть `Отец`, `Мать`, `Семья`, `Учреждение`, `Паспорт`, то они тоже должны приходить из общей базы
- любые изменения в одном модуле должны отражаться в связанных модулях автоматически

## 11. Первый приоритет на реализацию

Если идти по правильному порядку, то следующий рабочий этап должен быть таким:
