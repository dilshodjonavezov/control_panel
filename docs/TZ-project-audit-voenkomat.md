# ТЗ и аудит проекта «Военкомат»

## 1. Назначение проекта

Проект должен быть единой системой для военкомата и связанных ведомств:

- роддом
- ЗАГС
- ЖЭК
- паспортный стол
- школа
- колледж / вуз
- поликлиника
- ВВК
- пограничная служба
- админ

Главная цель: не просто хранить карточки граждан, а автоматически формировать воинский учет, основания для отсрочки и основания для освобождения по событиям из жизни гражданина.

Ключевой сценарий:

1. Ребенок рождается в роддоме.
2. Запись передается в ЗАГС.
3. Создается или обновляется гражданин.
4. Ребенок связывается с матерью, отцом и семьей.
5. Если ребенок мужского пола, он автоматически ставится на первичный воинский учет.
6. У отца автоматически пересчитывается количество детей.
7. Если у отца 2 и более детей и он является призывником, система автоматически переводит его в освобождение по семейным обстоятельствам.

## 2. Что уже реализовано

По текущему коду в проекте уже есть рабочая основа:

- Angular frontend и NestJS backend.
- Модули по ролям: военкомат, роддом, ЗАГС, ЖЭК, паспорт, школа, колледж, поликлиника, ВВК, граница, админ.
- Реестр граждан, семей, адресов, паспортов, учебных записей, медзаписей, ВВК, пересечений границы и воинских записей.
- В `backend/src/modules/zags/zags.service.ts` уже реализована связка рождения с гражданином, семьей и воинским учетом.
- В `backend/src/modules/military/military.service.ts` уже реализован автопересчет семейной льготы для отца по количеству детей.
- В `src/app/services/voenkomat-data.service.ts` уже есть агрегирование данных для dashboard, карточки гражданина и проверки оснований отсрочек.

## 3. Что недоработано и мешает проекту

### 3.1. Позиционирование и названия

- В части интерфейсов и старых блоков проект до сих пор выглядит как абстрактная админка.
- Названия ролей и экранов были не везде согласованы.
- Заголовок приложения был `ControlPanel`, а не предметное название.

### 3.2. Битая кодировка в интерфейсе

- Во многих Angular-файлах видны строки вида `Р...`.
- Это означает, что часть экранов сохранена или читалась с поврежденной кодировкой.
- Из-за этого интерфейс выглядит сырым и недоделанным.

### 3.3. Смешение реальных данных и заглушек

- Часть экранов уже работает от backend API.
- Часть экранов до сих пор собрана на локальных массивах, моках и тестовых данных.
- Особенно это заметно в управлении пользователями, доступами и ряде витринных страниц.

### 3.4. Несогласованные идентификаторы

- В проекте одновременно используются `citizenId` и `peopleId`.
- По смыслу это почти всегда одна и та же ссылка на гражданина.
- Для дальнейшей разработки это нужно унифицировать.

### 3.5. Нет единого статуса жизненного цикла призывника

Сейчас логика разбросана по разным сущностям. Не хватает единого статуса:

- допризывник
- призывник
- имеет отсрочку
- освобожден
- негоден
- служит
- завершил службу
- выбыл / умер / за границей

### 3.6. Нет отдельных рабочих модулей военкомата

Для реальной эксплуатации не хватает:

- модуля повесток
- модуля явок
- модуля решений призывной комиссии
- модуля призывных кампаний
- модуля уклонистов
- модуля отчетов по району / кварталу / кампании
- нормального журнала межведомственных уведомлений

### 3.7. Недостаточно автоматизаций

Нужно довести бизнес-правила до системного уровня:

- автоотсрочка по очной учебе
- автоснятие отсрочки при отчислении
- автопереопределение статуса после решения ВВК
- автопривязка к военкомату по адресу
- автоуведомление военкомата при изменении паспорта, адреса, учебы и выезде за границу

### 3.8. Нет нормального контроля качества

- В проекте нет достаточного покрытия тестами бизнес-логики.
- Нет обязательных проверок на регрессию для критичных сценариев рождения, семьи и воинского учета.

## 4. Идеальная модель системы

### 4.1. Главная сущность

Главная сущность системы: `Citizen`.

Все остальные модули должны ссылаться на нее по ID:

- рождение
- ЗАГС
- адрес
- паспорт
- семья
- учеба
- медкарта
- ВВК
- граница
- воинский учет

### 4.2. Семья

Сущность `Family` должна хранить:

- отца
- мать
- детей
- число детей
- число сыновей
- число дочерей
- количество детей, поставленных на воинский учет при рождении
- признак права отца на освобождение

### 4.3. Воинская запись

Сущность `MilitaryRecord` должна быть центром для статуса призывника:

- текущий военный статус
- основание отсрочки
- основание освобождения
- ВВК категория
- дата постановки
- дата снятия
- привязанный военкомат
- история решений

## 5. Обязательный функционал MVP

### 5.1. Для военкомата

- dashboard по призывникам
- реестр граждан и призывников
- карточка гражданина с полным таймлайном
- реестр отсрочек
- реестр освобождений
- реестр решений ВВК
- реестр выехавших за границу
- журнал изменений

### 5.2. Для межведомственных ролей

- роддом создает запись о рождении
- ЗАГС подтверждает акт и семейные связи
- ЖЭК обновляет адрес и состав семьи
- паспортный стол обновляет документы
- школа / вуз обновляют учебу
- поликлиника и ВВК обновляют медоснования
- погранслужба обновляет выезд и въезд

### 5.3. Автоматические правила

- при регистрации рождения создается гражданин
- при наличии родителей создается или обновляется семья
- мальчик ставится на первичный воинский учет
- отцу пересчитывается число детей
- при 2+ детях у призывника включается освобождение по семейным обстоятельствам
- при очной учебе включается отсрочка
- при отчислении отсрочка снимается
- при негодности по ВВК военный статус меняется автоматически

## 6. Что нужно реализовать следующим этапом

### Этап 1. Привести проект в рабочий вид

- исправить битую кодировку во всех пользовательских экранах
- убрать остатки абстрактной админки
- согласовать названия ролей и страниц
- довести все основные экраны до работы от backend, а не от моков

### Этап 2. Довести ключевую бизнес-логику

- унифицировать `citizenId` и `peopleId`
- ввести единый статус жизненного цикла гражданина для военкомата
- завершить автопривязку адреса, семьи, учебы и воинской записи
- ввести единый audit trail для автоматических решений

### Этап 3. Добавить реальные модули военкомата

- повестки
- явки
- призывные комиссии
- кампании призыва
- уклонисты
- отчеты по району и возрастам
- массовые уведомления

## 7. Критерии готовности

- После регистрации ребенка в ЗАГС у него есть гражданин, семья и корректные связи с родителями.
- Если ребенок мужского пола, создается первичная воинская запись.
- У отца автоматически пересчитывается число детей.
- При двух и более детях у отца-призывника автоматически меняется статус на освобождение по семейным обстоятельствам.
- Карточка гражданина показывает полную связанную историю.
- Все основные модули отображают нормальный русский текст без битой кодировки.
- Основные управленческие экраны работают от backend API.
- Все критичные бизнес-сценарии покрыты тестами.

## 8. Замечания по текущему состоянию

На текущем этапе проект нельзя считать завершенным продуктом. Основа уже есть, и важная часть логики военкомата действительно реализована, но интерфейс, связность данных, административные экраны и полнота модулей пока не дотянуты до уровня рабочей системы.

Следующий правильный шаг: по этому ТЗ пройтись по модулям по очереди и добить их до рабочего состояния, начиная с кодировки, витрин военкомата и экранов управления.
